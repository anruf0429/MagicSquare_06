## 1. Executive Summary

4×4 정수 격자에 빈 칸(값 `0`)이 정확히 두 곳 있을 때, 누락된 두 숫자를 row-major로 정의된 첫·둘째 빈 칸에 **고정된 두 가지 조합 순서**로 배치해 마방진 합(34)을 만족시키고, 성공 시 `int[6]` 포맷으로 반환하는 **로컬·순수 도메인** 시스템을 구축한다. 본 프로젝트는 알고리즘 경쟁이 아니라 **Invariant(불변조건)를 명시하고 테스트로 고정하는 사고**, **입력·출력 계약을 한글 한 줄이 아닌 검증 가능한 조항으로 고정하는 설계**, **Dual-Track TDD(경계 RED / 도메인 RED 분리)**를 반복해 체득하는 것이 핵심 목적이다. 동일 요구를 **경계 UX Contract·도메인 Logic Rule 언어(§2.3)**와 **CI 품질 게이트(§7.1)**로도 고정한다.

## 2. Problem Statement (문제 정의)

### 2.1 문제의 초점

표면적 목표인 “마방진을 만든다”가 아니라, **주어진 상태가 마방진 불변조건을 만족하는지를 검증 가능한 규칙으로 완성·판별한다**는 관점이 본 PRD의 문제 정의다. 완성 결과는 **항상 동일한 계약(형·값·순서)**으로 표현되어야 하며, 실패는 **원인 코드와 고정 메시지**로 구분되어야 한다.

### 2.2 입력·출력 계약이 핵심인 이유

- **검증 자동화**: 계약이 숫자·형·개수·좌표 체계까지 고정되어야 테스트가 실패 원인을 단언할 수 있다.
- **회귀 안정성**: 리팩토링 후에도 동일 입력에 동일 출력·동일 오류 코드가 나와야 “동작 보존”을 증명할 수 있다.
- **경계·도메인 분리**: 경계에서 걸러야 할 오류와 도메인에서만 알 수 있는 실패(예: 두 조합 모두 비마방진)를 혼동하면 테스트가 거짓 양성·거짓 음성을 낳는다. 계약은 두 레이어의 책임 경계를 고정한다.

### 2.3 Dual-Track 언어: UX Contract · Logic Rule

본 PRD에서 **슬라이드의 “UI 트랙”은 그래픽 화면이 아니라**, 호출자·테스트가 관측하는 **경계(Boundary) 출력 계약**이다. `UI_*` 코드는 **픽셀 UI가 아니라 API·CLI·테스트가 보는 오류·성공 표현**을 가리킨다.

| UX Contract (관측 가능한 “경계 상태”) — 예시 언어 | 본 PRD에서의 대응 |
|---------------------------------------------------|-------------------|
| 표시됨 / 표시 안 됨 | 고정 `message`·성공 페이로드(`int[6]`) 노출 여부 |
| 가능 / 불가 | 경계 실패 시 도메인 솔버 호출 불가(FR-01-AC-05) |
| 포함 / 미포함 | 실패 응답에 성공 스키마 필드 미포함(PROP-04) |

| Logic Rule (도메인 동작) — 예시 언어 | 본 PRD에서의 대응 |
|-------------------------------------|-------------------|
| 허용 / 거부 | 입력 유효 시 솔버 진입; `DOMAIN_*`·해 없음 판정 |
| 계산 / 반환 | 빈칸·누락 수·마방진 판정·두 조합 시도 후 `int[6]` 반환 |
| 차단 | BR 위반 시 해당 단계에서 실패 코드 |

**테스트 후보 선별**: 위 언어로 **이진 결정(성공/실패·코드 한 가지)**이 서술된 시나리오만 RED 테스트로 옮긴다. 범위 선언만 있고 관측 가능한 결과가 없는 할 일은 테스트 ID를 부여하지 않는다.

## 5. Functional Requirements (기능 요구사항)

공통 **입력 타입(경계 통과 후 도메인으로 전달되는 값)**: `int[][]`, 행 4·각 행 길이 4. **출력 타입(성공)**: `int[6]` = `[r1,c1,n1,r2,c2,n2]`, 좌표는 **1-index**.

---

### FR-01 입력 검증 (Boundary)

| 필드 | 내용 |
|------|------|
| **Feature ID** | FR-01 |
| **설명** | 호출자가 전달한 행렬이 §6 Business Rules 및 BR 경계 규칙을 만족하는지 검증한다. 실패 시 도메인 솔버를 호출하지 않는다. |
| **입력** | `int[][] matrix` (미검증 원시 입력). |
| **처리 규칙** | BR-01~BR-05를 순서 무관하게 검사한다. 한 규칙이라도 위반하면 즉시 실패로 종료한다. 불변조건: 경계는 도메인 타입을 생성하지 않거나, 생성하더라도 실패 시 도메인 로직에 진입하지 않는다. |
| **출력** | 성공: 검증 통과 플래그 또는 도메인에 넘길 수 있는 동치 구조. 실패: `UI_*` 고정 코드 및 고정 `message` (`report/02` §2.2 테이블). |
| **승인 기준 (AC)** | **FR-01-AC-01** 행 수≠4 또는 어떤 행 길이≠4이면 결과 코드=`UI_INVALID_SIZE`, 메시지=`matrix must be 4x4`. **FR-01-AC-02** `0` 개수≠2이면 `UI_INVALID_BLANK_COUNT`, `matrix must contain exactly two zeros`. **FR-01-AC-03** 원소가 0 또는 1~16 밖이면 `UI_OUT_OF_RANGE`, `values must be 0 or in [1,16]`. **FR-01-AC-04** 0 제외 중복 존재 시 `UI_DUPLICATE_NONZERO`, `non-zero values must be unique`. **FR-01-AC-05** 위 실패 케이스에서 도메인 `solve`/판정 함수 호출 횟수=0(스파이·모킹으로 검증). |
| **관련 오류/예외 정책** | 경계 전용 코드 `UI_INVALID_SIZE`, `UI_INVALID_BLANK_COUNT`, `UI_OUT_OF_RANGE`, `UI_DUPLICATE_NONZERO`. 메시지 문자열은 위 AC와 **완전 일치**해야 한다. |

---

### FR-02 빈칸 탐색

| 필드 | 내용 |
|------|------|
| **Feature ID** | FR-02 |
| **설명** | 값이 `0`인 두 칸의 좌표를 row-major 순서로 반환한다. |
| **입력** | FR-01을 통과한 4×4 행렬. |
| **처리 규칙** | BR-06~BR-07. 첫 빈칸=row-major 첫 `0`, 둘째=그 다음 `0`. |
| **출력** | 순서쌍 두 개 `(r1,c1)`, `(r2,c2)`, 각각 1-index, 범위 1~4. |
| **승인 기준 (AC)** | **FR-02-AC-01** 입력에 `0`이 정확히 2개일 때 좌표 개수=2. **FR-02-AC-02** 반환 순서가 row-major와 동일함을 예상 좌표와 비교한다. **FR-02-AC-03** 모든 좌표 정수 ∈[1,4]. |
| **관련 오류/예외 정책** | 빈칸 개수≠2는 FR-01에서 차단. 도메인만 호출되는 경로에서 재검증 시 `DOMAIN_INVALID_BLANK_COUNT`. |

---

### FR-03 누락 숫자 탐색

| 필드 | 내용 |
|------|------|
| **Feature ID** | FR-03 |
| **설명** | 격자에 나타나지 않은 1~16의 원소 두 개를 찾아 오름차순으로 `small`, `large`로 정한다. |
| **입력** | FR-01 통과 4×4 행렬. |
| **처리 규칙** | BR-08. multiset `{1..16}`에서 등장한 비0 값을 제외한 나머지가 정확히 2개여야 한다. |
| **출력** | `small` ≤ `large`인 두 정수. |
| **승인 기준 (AC)** | **FR-03-AC-01** 누락 집합 크기=2가 아니면 `DOMAIN_INVALID_MISSING_COUNT`. **FR-03-AC-02** 누락이 2개일 때 `small`=`min(누락)`, `large`=`max(누락)`. |
| **관련 오류/예외 정책** | `DOMAIN_INVALID_MISSING_COUNT`. |

### FR-04 마방진 판정

| 필드 | 내용 |
|------|------|
| **Feature ID** | FR-04 |
| **설명** | 0이 없는 완전 채워진 4×4 정수 행렬에 대해 마방진 상수 34를 만족하는지 판정한다. |
| **입력** | 4×4 `int[][]`, 모든 셀 1~16(검증된 완전 행렬). |
| **처리 규칙** | BR-09~BR-12. |
| **출력** | `true` 또는 `false`. |
| **승인 기준 (AC)** | **FR-04-AC-01** 알려진 마방진 해면 `true`. **FR-04-AC-02** 합이 34가 아닌 행·열·대각이 하나라도 있으면 `false`. **FR-04-AC-03** 판정은 입력 행렬 비변조 전제에서 반복 실행 시 동일 결과. |
| **관련 오류/예외 정책** | 입력이 완전하지 않거나 범위 위반이면 선행 검증 실패(FR-01 또는 도메인 검증)로 처리; FR-04 자체는 “완전·유효한 행렬” 전제. |

### FR-05 해 찾기(solution): 두 조합 시도 및 반환

| 필드 | 내용 |
|------|------|
| **Feature ID** | FR-05 |
| **설명** | 누락 두 숫자를 첫·둘째 빈 칸에 넣는 두 가지 고정 순서를 시도하고, 처음 성공하는 배치를 `int[6]`으로 반환한다. 둘 다 실패하면 §7 및 BR-13에 따른 도메인 실패를 반환한다. |
| **입력** | FR-01 통과 4×4 행렬. |
| **처리 규칙** | BR-13~BR-16. 시도1: `small`→첫 빈칸, `large`→둘째 빈칸. 시도2: `large`→첫 빈칸, `small`→둘째 빈칸. 각 시도 후 FR-04로 마방진 검사. |
| **출력** | 성공: `[r1,c1,n1,r2,c2,n2]` — `(r1,c1)`는 첫 빈칸 1-index, `(r2,c2)`는 둘째 빈칸 1-index, `n1`·`n2`는 각각 첫·둘째 빈칸에 배치된 값. 실패: `DOMAIN_NOT_MAGIC`; 경계 표현 시 `UI_DOMAIN_FAILURE` 및 고정 메시지 `no valid magic-square completion found`. |
| **승인 기준 (AC)** | **FR-05-AC-01** 성공 시 배열 길이=6. **FR-05-AC-02** 성공 시 모든 좌표 1-index ∈[1,4]. **FR-05-AC-03** 시도1이 마방진이면 시도2를 수행하지 않고 시도1 배치를 반환한다. **FR-05-AC-04** 시도1 실패·시도2 성공 시 반환 값이 시도2 배치와 일치한다. **FR-05-AC-05** 두 시도 모두 실패 시 코드=`DOMAIN_NOT_MAGIC`(도메인), 경계 매핑 시 `UI_DOMAIN_FAILURE` 및 위 메시지. **FR-05-AC-06** 성공 행렬의 모든 행·열·대각선 합=34. |
| **관련 오류/예외 정책** | 도메인: `DOMAIN_NOT_MAGIC`. 경계 출력: `UI_DOMAIN_FAILURE`. 입력 오류는 FR-01 선행. |

## 6. Business Rules (도메인 규칙)

“항상 참이어야 하는 규칙”으로 서술한다.

| ID | 규칙 문장 |
|----|-----------|
| BR-01 | 입력 행렬의 행 개수는 4이고, 각 행의 길이는 4이다. |
| BR-02 | 모든 셀 값은 0이거나 1 이상 16 이하의 정수이다. |
| BR-03 | 값이 0인 셀은 정확히 두 개이다. |
| BR-04 | 0이 아닌 값들은 서로 중복되지 않는다. |
| BR-05 | BR-01~BR-04를 동시에 만족하지 않는 입력은 유효하지 않다. |
| BR-06 | 첫 번째 빈 칸은 row-major 순회에서 가장 먼저 만나는 0의 위치이다. |
| BR-07 | 두 번째 빈 칸은 첫 번째 빈 칸 이후 row-major 순회에서 만나는 유일한 나머지 0의 위치이다. |
| BR-08 | 집합 `{1,…,16}` 중 격자의 비0 값으로 등장하지 않은 정수는 정확히 두 개이며, 그 둘이 누락 숫자이다. |
| BR-09 | n=4 마방진의 마방진 상수는 34이다. |
| BR-10 | 마방진 판정에서 네 행의 각 행합은 34이다. |
| BR-11 | 마방진 판정에서 네 열의 각 열합은 34이다. |
| BR-12 | 마방진 판정에서 주대각선 합과 부대각선 합은 각각 34이다. |
| BR-13 | 누락 두 수 중 작은 값을 `small`, 큰 값을 `large`라 한다. |
| BR-14 | 시도 1은 `small`을 첫 번째 빈 칸에, `large`를 두 번째 빈 칸에 배치한다. |
| BR-15 | 시도 1의 결과가 마방진이 아니면 시도 2에서 `large`를 첫 번째 빈 칸에, `small`을 두 번째 빈 칸에 배치한다. |
| BR-16 | 시도 1이 마방진이면 시도 2를 적용하지 않는다. |
| BR-17 | 성공 시 반환 배열은 `[r1,c1,n1,r2,c2,n2]`이며, `(r1,c1)`·`(r2,c2)`는 각각 첫·둘째 빈 칸의 **1-index** 좌표이고, `n1`·`n2`는 해당 칸에 놓인 값이다. |
| BR-18 | 시도 1과 시도 2 모두 마방진이 아니면 해가 없다고 판정한다. |

| NFR-03 | **결정론**: 동일 입력에 대해 동일 성공 배열 또는 동일 오류 코드·메시지. | 반복 실행 테스트. |
| NFR-04 | **입력 행렬 비변조**: FR-01~FR-05 경로에서 호출자가 넘긴 원본 `int[][]`를 **변경하지 않는다**(필요 시 내부에서 복사본으로 연산). 테스트: 호출 전후 깊은 동등성 또는 변경 금지 단언. |
| NFR-05 | **성능(선택)**: 4×4 단일 요청 처리 시간 **50ms 이내**(동일 프로세스·개발용 하드웨어 기준). 범위 초과 시 성능 최적화가 아니라 요구 완화 여부를 문서 변경으로 결정한다. | 벤치마크 또는 타이밍 단언(선택). |
| NFR-06 | **CI 품질 게이트**: NFR-01·NFR-02 하한 미달 시 파이프라인 실패로 처리한다. | CI 로그·커버리지 리포트 아티팩트. |

### 7.1 CI · 산출물 (MLOps 정렬)

본 프로젝트는 **모델 학습 파이프라인이 없으며**, MLOps와의 정렬은 **계약·결정론·품질 게이트·재현 가능한 산출물**로 한다.

| 단계(권장 순서) | 목적 |
|-----------------|------|
| 정적 검사·포맷 | 코드 일관성 |
| 도메인 단위 테스트 (Track B) | Logic Rule·`DOMAIN_*` 고정 |
| 경계·통합 테스트 (Track A·IT) | UX Contract·`UI_*`·호출 순서 |
| 커버리지 산출 | NFR-01·NFR-02 만족 여부 판정 |

**아티팩트**: 커버리지 리포트(예: HTML/XML), 테스트 결과 로그. 선택적으로 성능(NFR-05) 측정 결과를 같은 파이프라인에 붙일 수 있다.

## 8. Dual-Track TDD Strategy

### 8.0 트랙 정의 (요약)

- **Track A (Boundary / “UI”)**: **UX Contract 기반 RED** — 고정 `code`·`message`·성공 시 `int[6]` 스키마가 설계대로 보이는지 검증한다.
- **Track B (Domain / Logic)**: **Logic Rule 기반 RED** — BR·FR-02~05·`DOMAIN_*`가 규칙대로인지 검증한다.
- **원칙**: UI는 디자인(여기서는 **경계 출력 계약**)으로 RED를 만들고, 로직은 규칙으로 RED를 만든다. **GREEN과 REFACTOR는 두 트랙을 함께 통과한 뒤에 진행**한다(§8.3).

### 8.3 병렬 진행 규칙

1. **UI RED**와 **Logic RED**를 동시에 유지 가능한 범위에서 병렬로 추가한다.  
2. **UI GREEN**과 **Logic GREEN**은 각 트랙에서 최소 구현으로 통과시킨다.  
3. **REFACTOR**는 두 트랙 테스트가 모두 통과한 상태에서만 수행한다.  
4. **금지**: “도메인을 전부 구현한 뒤 경계 검증을 나중에 추가”하여 경계 실패 시 도메인이 이미 호출되는 구조로 가는 것. 경계 선행 실패는 항상 도메인 미호출을 만족해야 한다.  
5. 할 일을 테스트 케이스로 바꿀 때 **§2.3**의 테스트 후보 선별 규칙을 따른다.

### 8.4 Boundary와 Domain 검증 진입 (정책)

- **제품·통합 경로(권장)**: 호출자는 원시 `int[][]`를 **Boundary(FR-01)**에 넘긴다. FR-01 실패 시 도메인 솔버는 호출하지 않는다.  
- **도메인 단위 테스트(Track B)**: 동일한 불변 조건을 도메인 API에 직접 넣어 **`DOMAIN_*` 코드**로 RED를 만들 수 있다. 이는 “경계를 거치지 않은 진입”을 전제로 한 테스트이며, 통합 경로의 순서·매핑과 모순되면 안 된다.  
- **구현 선택(택1·문서·코드에서 하나로 통일)**: (a) 솔버는 “FR-01을 통과한 입력만 받는다”고 가정하고 내부 재검증을 생략하거나, (b) 솔버 진입 시 도메인에서 동일 규칙을 한 번 더 검증한다. 선택한 쪽은 **통합 테스트(IT-*)**가 전제와 호출 횟수를 검증한다.

자세한 논리 컴포넌트·계약은 `docs/DESIGN_layered_architecture_tdd_magic_square_4x4.md`와 정합시킨다.

### 9.4 테스트 데이터 — 대표 4×4 행렬

**데이터셋 A (공통 Given — 시도·역시도 검증에 사용)**

| r\\c | c1 | c2 | c3 | c4 |
|------|----|----|-----|-----|
| r1 | 16 | 2 | 3 | 13 |
| r2 | 5 | 11 | 10 | 8 |
| r3 | 9 | 7 | **0** | 12 |
| r4 | 4 | 14 | 15 | **0** |

- **기대(탐색)**: 첫 빈칸 1-index=(3,3), 둘째=(4,4)(문서 기호 r1~4 기준). 구현 언어가 0-index 내부 사용 시 테스트만 1-index로 단언한다.

**데이터셋 B (입력 오류 샘플 — 예시)**

| 목적 | 조작 |
|------|------|
| 빈칸 1개 | 위 행렬에서 한 곳의 0을 1로 채우고 다른 0 유지 |
| 중복 | 비0 칸 하나를 다른 비0과 동일 값으로 변경 |
| 범위 초과 | 임의 한 칸을 17로 변경 |

### 9.5 Property / Invariant 기반 체크 항목

| PROP-02 | 성공 시 `n1`, `n2`는 데이터셋의 누락 두 수이며 위치와 일치. |

## 10. Architecture Overview (High-Level)

### 10.1 레이어 정의

| 레이어 | 책임 |
|--------|------|
| **Boundary** | 원시 입력 검증(FR-01), 출력 포맷·오류 코드 매핑, 도메인 실패→`UI_DOMAIN_FAILURE` 변환. |
| **Domain** | FR-02~FR-05 순수 규칙, 마방진 판정, 조합 시도. 외부 IO·문자열 포맷(UI 메시지)에 의존하지 않는다. |

### 10.2 책임 분리(SRP)와 확장(OCP)

- **SRP**: 빈칸 탐색·누락 수 계산·판정·조합 시도를 별 책임으로 두어 테스트 이름과 1:1에 가깝게 유지한다.  
- **OCP**: N×N 확장은 기존 4×4 테스트를 깨지 않는 **별도 어댑터/전략 추가**로만 허용한다(본 PRD Out-of-Scope이지만 원칙만 명시).

### 10.3 의존성 방향

- **도메인은 Boundary를 알지 못한다.** Boundary만 도메인 API를 호출한다(ECB: `boundary → control → entity`, entity는 상위 비의존). 솔버 내부 재검증 여부는 **§8.4**에서 택1로 고정한다.

## 11. Risks & Ambiguities

### 11.1 결정 항목(모호성 제거)

| 결정 ID | 내용 |
|---------|------|
| DEC-01 | 두 시도 모두 실패 시 도메인 코드=`DOMAIN_NOT_MAGIC`, 경계 표시=`UI_DOMAIN_FAILURE`, 메시지=`no valid magic-square completion found`. |
| DEC-02 | 좌표는 **항상 1-index**로 반환; 구현 내부 0-index 사용 가능하나 경계 노출은 변환 후 검증. |

| DEC-05 | 통합 경로에서 **Boundary(FR-01) 선행**은 불변이다. 도메인 단독 진입 시 검증을 한 번 더 할지는 §8.4 (a)/(b) 중 **하나로 통일**하고 IT로 검증한다. |
